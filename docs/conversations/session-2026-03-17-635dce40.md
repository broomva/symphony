> **Plan: Symphony CLI**

> **Context**
> 
> Symphony currently has a minimal CLI (`symphony [WORKFLOW.md] [--port]`) that only starts the daemon. There is no way to query, control, or test Symphony without raw `curl` commands or external scripts. This plan creates a comprehensive CLI that acts as both daemon launcher and client — the primary developer interface for Symphony. It leverages the existing HTTP API, integrates with the control metalayer, and follows Rust CLI best practices with clap v4 derive.
> 
> **Subcommands**
> 
> ```
> symphony                              # start daemon (backward compat)
> symphony start [WORKFLOW.md] [--port] # start daemon (explicit)
> symphony stop                         # graceful shutdown via API
> symphony status                       # query running daemon state
> symphony issues                       # list running + retrying issues
> symphony issue            # detail for one issue
> symphony refresh                      # trigger immediate poll
> symphony workspaces                   # list workspace directories
> symphony workspace  [--clean]     # show/manage a workspace
> symphony validate [WORKFLOW.md]       # validate workflow without starting
> symphony config [WORKFLOW.md]         # display resolved config
> symphony check                        # run make smoke equivalent
> symphony audit                        # full control-audit
> symphony test [--crate ]        # run tests with filtering
> symphony logs [--follow] [--id X]     # tail daemon log file
> ```
> 
> All client commands (status, stop, issues, issue, refresh, workspaces) accept `--port` globally or read `SYMPHONY_PORT` env var. Default: 8080.
> 
> **File Layout**
> 
> ```
> src/
>   main.rs             → entry point: parse args, route to command
>   cli/
>     mod.rs            → Cli struct + Command enum (clap derive)
>     start.rs          → start command (extracted from current run())
>     client.rs         → SymphonyClient (HTTP client to daemon)
>     status.rs
